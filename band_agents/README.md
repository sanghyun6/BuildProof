# BuildProof Band Audit Court — Sidecar Agents

Four Python agents that run as **Band External Agents** and deliberate over a
BuildProof audit report inside a shared Band room. The Next.js web app
generates the Audit Court packet locally; Band becomes the agent-to-agent
coordination layer when these sidecar agents are running and joined to the
room.

## What is automated vs. still manual

| Step | Automation |
|------|-----------|
| Audit pipeline (claim → evidence → verdict → score) | Fully automated inside the BuildProof web app |
| Audit Court packet generation | Fully automated — generated by `lib/bandCourtPacket.ts` |
| Posting the packet into a Band room | **Manual** — user copies the combined message and pastes it into the room |
| Multi-agent deliberation in the room | Fully automated — the four sidecar agents reply to one another via @-mentions |
| Final Consensus Verdict | Posted in the room by `@BuildProofLeadJudge` after specialists reply |

The web app does **not** make an outbound REST/WS call to Band. The bridge
between BuildProof and Band is the packet you paste into the room.

## Agents

| Script | Default handle | Role |
|--------|----------------|------|
| `lead_judge.py` | `@BuildProofLeadJudge` | Convenes the panel, posts the final consensus verdict |
| `claim_prosecutor.py` | `@BuildProofClaimProsecutor` | Challenges weak / README-only claims |
| `evidence_defender.py` | `@BuildProofEvidenceDefender` | Defends claims using only the listed evidence |
| `repo_forensics.py` | `@BuildProofRepoForensics` | Classifies each repository signal |

Each script is a thin wrapper around `_runtime.py`, which loads credentials
via the official `band.config.load_agent_config` helper, builds an
`AnthropicAdapter`, and starts an `Agent` connected to the Band websocket.

## Prerequisites

- Python 3.11+
- An Anthropic API key (`ANTHROPIC_API_KEY`)
- A Band account
- Four Band **External Agents** created in the Band dashboard (one per role)

## Setup

### 1. Install Python dependencies

```bash
cd band_agents
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Or with [uv](https://docs.astral.sh/uv/):

```bash
cd band_agents
uv venv
uv pip install "band-sdk[anthropic]" python-dotenv
```

The `band-sdk[anthropic]` extra pulls in the Anthropic SDK, `python-dotenv`,
and `pyyaml` automatically.

### 2. Create four External Agents on Band

1. Open [app.band.ai](https://app.band.ai) and sign in.
2. Open the agent dashboard and click **Create External Agent** four times.
3. Name them so it is obvious which role each one plays — suggested names:
   - `BuildProofLeadJudge`
   - `BuildProofClaimProsecutor`
   - `BuildProofEvidenceDefender`
   - `BuildProofRepoForensics`
4. For each agent, copy the **agent_id (UUID)** and **API key** Band shows you.
   The API key is shown only once — store it safely.

> Display names in Band are workspace-scoped. The packet's @-mentions assume
> the four names above. If you used different names, edit your local copy of
> `lib/bandCourtPacket.ts` or adjust the @-handles before pasting the packet.

### 3. Paste credentials into `agent_config.yaml`

```bash
cp agent_config.yaml.example agent_config.yaml
```

Open `agent_config.yaml` and fill in the four blocks:

```yaml
lead_judge:
  agent_id: "uuid-band-gave-you-for-lead-judge"
  api_key:  "band-api-key-for-lead-judge"
  handle:   "@yourhandle/buildproofleadjudge"

# ...repeat for claim_prosecutor, evidence_defender, repo_forensics
```

`agent_config.yaml` is in the repo's `.gitignore` — it is never committed.

### 4. Set your Anthropic API key

Either export `ANTHROPIC_API_KEY` in your shell, or put it in a `.env` file
inside this directory (`band_agents/.env`, also gitignored):

```bash
echo "ANTHROPIC_API_KEY=sk-ant-..." > .env
```

## Running the agents

### Option A — all four agents in one process (simplest)

```bash
cd band_agents
python run_all.py
```

`run_all.py` boots all four agents concurrently via `asyncio.gather`. Each
agent maintains its own Band websocket. Press `Ctrl+C` to stop them all.

### Option B — each agent in its own terminal (best for debugging)

```bash
cd band_agents
python lead_judge.py            # Terminal 1
python claim_prosecutor.py      # Terminal 2
python evidence_defender.py     # Terminal 3
python repo_forensics.py        # Terminal 4
```

Each script prints its display name as it starts and again once the websocket is up:

```
[BuildProofLeadJudge] starting — attempting Band connection as 'lead_judge'...
[BuildProofLeadJudge] running — waiting for Band messages.
```

If `agent_config.yaml` is missing, the script exits immediately with a clear
message pointing at `agent_config.yaml.example`. If `ANTHROPIC_API_KEY` is
missing, the script exits before any Band connection is attempted. Secrets
are never printed.

## Running the Audit Court live

1. Run a BuildProof audit in the web app (`npm run dev` in the project root).
2. After the report loads, expand the **Band Audit Court** section.
3. Click **Copy combined Band room message** — that puts the @-mentions plus
   the structured packet onto your clipboard.
4. Start all four sidecar agents (see above).
5. Open a Band room and add all four agents as members.
6. (Recommended) First send the **Band agent self-test message** described
   below to confirm every agent is actually present in the room.
7. Paste the combined court message into the room.

The packet opens with @-mentions for all four agents. `@BuildProofLeadJudge`
convenes the court and the three specialists each respond in turn via Band's
own @-mention routing. The Lead Judge then posts the **Final Consensus
Verdict**.

### Band room self-test (presence check)

Before running the full court packet, verify all four sidecars are joined to
the room and live. Either click **Copy Band agent self-test message** in the
web app's Band Audit Court panel, or paste this line into the Band room
directly:

```
@BuildProofLeadJudge @BuildProofClaimProsecutor @BuildProofEvidenceDefender @BuildProofRepoForensics Please each reply READY with your role name.
```

You should see four short replies — one per agent:

```
READY — BuildProofLeadJudge
READY — BuildProofClaimProsecutor
READY — BuildProofEvidenceDefender
READY — BuildProofRepoForensics
```

If any of the four does not reply within ~30 seconds, that sidecar is not
running, not in the room, or has bad credentials. Fix it (re-run
`python check_band_setup.py` and `python smoke_band_agents.py`, then start
the missing process) before sending the combined court message.

## How coordination works

Band routes messages between agents by `@-mention`. The opening message in
the room calls all four agents explicitly:

```
@BuildProofLeadJudge — please convene the BuildProof Audit Court for the packet below.

Coordinate the review with the three specialists in this Band room:
  @BuildProofClaimProsecutor — challenge weak or unsupported claims.
  @BuildProofEvidenceDefender — defend claims using only the listed evidence.
  @BuildProofRepoForensics — classify each piece of repository evidence.
```

Each agent's role-specific prompt (in `prompts/`) instructs it to wait for an
@-mention from the Lead Judge and to address replies back to
`@BuildProofLeadJudge`. The Lead Judge posts the final consensus once all
three specialists have replied.

## Customization

- **System prompts** live in `prompts/` as one Markdown file per role
  (`lead_judge.md`, `claim_prosecutor.md`, `evidence_defender.md`,
  `repo_forensics.md`). Edit these to change agent voice or rules.
- **Model**: `_runtime.DEFAULT_MODEL` selects Claude Sonnet by default. Pass
  a different model when constructing the agent if you need to override it.
- **Display handles** in the packet text live in `lib/bandCourtPacket.ts`
  under `BAND_AGENT_HANDLES`.

## Honest claim summary

What we can claim:

- BuildProof uses **Band as its agent-to-agent coordination layer** for the
  Audit Court review when the four sidecar agents are running.
- The four agents are real Band External Agents using the official `band`
  Python SDK (`AnthropicAdapter`, `Agent.create`, `agent.run()`).
- The web app generates a structured packet automatically; the manual paste
  step is the bridge between the audit pipeline and Band.

What we should **not** claim:

- That the web app talks to Band directly. It does not — the bridge is a
  copy-paste of the packet.
- That deliberation runs without the sidecar agents. Without them, the
  pasted packet is just text; nothing replies.

## Known issues

- The Band SDK is a moving target. Constructor arguments for
  `AnthropicAdapter` (currently `model`, `provider_key`, `prompt`) and
  `Agent.create` (currently `adapter`, `agent_id`, `api_key`) should be
  verified against the version of `band-sdk` pinned in `requirements.txt`.
- Each agent is restartable independently; agents do not share state across
  restarts beyond what Band itself persists.
